# 游戏发行后台市场、渠道与同步规则设计

## 背景与范围

本文档用于补全现有中文架构文档中关于 `market`、渠道实例、模板版本、支付路由、`sandbox -> production` 同步以及前端交互的关键规则。

本文档仅基于以下文档和本轮确认结论整理：

- `docs/architecture/zh-CN/README.md`
- `docs/architecture/zh-CN/go_domain_api_draft.md`
- `docs/architecture/zh-CN/backend_agent_execution.md`
- `docs/architecture/zh-CN/frontend_agent_execution.md`

本文档目标不是重写全部架构，而是把当前最容易产生歧义、最影响后端 API 与前端信息架构的规则钉死，供后续写回架构文档和拆实现计划使用。

## 已确认结论

- 一个游戏可以同时启用多个 `market`
- `market` 固定枚举为：`GLOBAL / JP / KR / SEA / HMT / CN`
- 默认市场为 `GLOBAL`
- `market` 表示发行大区，不等于国家
- `GLOBAL` 表示默认兜底海外市场
- 具体海外 `market` 优先于 `GLOBAL`
- `CN` 只显示国内渠道
- 非 `CN` 市场只显示非国内渠道
- 渠道配置按 `market` 维度拆分，不是游戏级共用一套
- 因市场规则变化而不兼容的渠道实例，保留配置但标红
- 管理员可手动隐藏不兼容渠道实例
- 被隐藏的渠道实例完全失效：不进入快照、不参与同步、不出现在客户端最终配置、不出现在默认列表
- 支付路由空字段等于 `*`
- 同一游戏、同一 `pay_way` 下，生效中的 `priority` 不允许重复
- 同一游戏、同一 `pay_way` 下，归一化后的路由选择器不允许重复
- 模板驱动配置由后台维护，且必须版本化管理
- 模板版本生命周期固定为 `draft / published / archived`
- 同步按 `section` 级执行，支持部分选择

## Market 与渠道实例模型

### Market 定义

系统当前固定使用以下 `market` 枚举：

- `GLOBAL`
- `JP`
- `KR`
- `SEA`
- `HMT`
- `CN`

其中：

- `GLOBAL` 表示默认兜底海外市场
- `CN` 只代表中国大陆市场
- `JP / KR / SEA / HMT` 代表具体海外发行大区
- 当具体海外 `market` 与 `GLOBAL` 同时存在时，具体 `market` 优先于 `GLOBAL`

### 渠道可见性规则

- `market = CN` 时，只允许选择国内渠道
- `market != CN` 时，只允许选择非国内渠道
- `GLOBAL` 仅显示非国内渠道
- 新增渠道时必须按目标 `market` 过滤候选渠道
- 服务端必须做同样的兼容性校验，不能只依赖前端过滤

### 渠道实例模型

渠道配置不是游戏级共享一套，而是按 `market` 维度独立维护。

建议领域关系为：

- `Game`：负责游戏基础信息、已启用市场集合、法务链接、自有账号认证、商品、收银台绑定
- `GameMarket`：负责某个游戏下某个 `market` 的生效视角
- `GameMarketChannel`：负责某个 `market` 下的渠道实例、隐藏状态、渠道配置、渠道包、渠道登录、IAP 配置

同一个游戏下，不同 `market` 的同一渠道属于不同实例。例如：

- `GLOBAL / Google`
- `JP / Google`

它们必须被视为两条独立记录，而不是一条记录绑定多个 `market`。

### 不兼容与隐藏

如果市场变更导致某个渠道实例与当前 `market` 规则不兼容：

- 不自动删除
- 保留配置
- 在管理界面标红
- 明确标记为“不兼容当前 market”

管理员可对该实例执行手动隐藏。隐藏后的渠道实例必须视为完全失效数据：

- 不进入默认列表
- 不进入配置快照
- 不参与同步预览
- 不参与同步执行
- 不进入客户端最终配置

隐藏后的实例仍需保留记录，供审计和恢复使用。

## 渠道模板复用与实例复制

### 模板定义可复用

同一逻辑渠道在不同 `market` 下可以复用同一套模板定义，具体包括：

- `form_schema_json`
- `validation_rules_json`
- `secret_fields_json`
- `file_fields_json`

### 实际配置必须独立

不同 `market` 下的渠道实例，其实际配置数据必须独立维护，至少包括：

- 登录参数
- 密钥
- 文件
- IAP 配置
- 包绑定
- 配置状态
- 隐藏状态

因此，`JP / Google` 与 `GLOBAL / Google` 可以共用模板，但不能共用配置实例。

### 新增时复制初始值

新增某个 `market` 的渠道实例时，支持从其他 `market` 的同渠道复制初始值。

复制规则固定为：

- 仅复制普通字段
- `secret` 字段必须清空
- `file` 字段必须清空
- 复制只发生在创建时
- 创建完成后，新旧实例不再联动

复制创建后，新实例初始状态固定为：

- `config_status = invalid`

原因是普通字段虽然已经带入，但敏感字段或文件字段仍未补齐，配置不能视为可用。

此时：

- `last_check_message` 必须明确提示“缺少必填敏感字段或文件字段”
- 管理员补齐必填 `secret` / `file` 字段并通过校验后，状态才允许转为 `valid`

## 模板版本生命周期

所有后台可编辑模板都必须做版本化管理，版本生命周期固定为：

- `draft`
- `published`
- `archived`

### draft

`draft` 表示草稿版本：

- 可编辑
- 可调整字段
- 可修改价格行
- 可做预览校验
- 不得直接生效

`draft` 不允许：

- 进入客户端最终配置
- 进入正式配置快照
- 参与 `sandbox -> production` 同步执行

### published

`published` 表示当前正式生效版本：

- 可被游戏绑定
- 可参与配置快照
- 可参与同步
- 可进入客户端最终配置

规则如下：

- 同一模板任一时刻仅允许存在一个 `published` 版本
- `published` 版本一旦生成后应视为只读
- 不允许直接原地编辑 `published` 版本
- 如需修改，必须基于新草稿版本调整后重新发布

### archived

`archived` 表示历史归档版本：

- 只读保留
- 用于审计
- 用于追溯
- 用于差异对比
- 用于历史回看

`archived` 版本不得：

- 被新游戏继续绑定
- 直接重新发布

如需复用，必须复制为新的 `draft` 版本后再发布。

### 状态流转规则

允许的状态流转只有：

- `draft -> published`
- `published -> archived`

当同一模板发布新的 `published` 版本时：

- 旧的 `published` 版本必须自动转为 `archived`

不允许：

- `archived -> published` 直接回退
- 跳过 `draft` 直接生成新的正式版本

## Section 级同步规则

同步以 `section` 为最小执行单位，支持部分选择，不要求整游戏全量同步。

当前固定 `section` 枚举为：

- `game`
- `markets`
- `legal`
- `channels`
- `packages`
- `products`
- `cashier`
- `payments`
- `config`

后续可以扩展新的 `section`，但当前版本只允许上述枚举值参与预览和执行；未识别的 `section` 必须拒绝受理。

### 预览规则

`sync/preview` 必须按 `section` 返回差异结果。

前端必须按 `section` 展示：

- 新增
- 修改
- 删除

并允许管理员按 `section` 勾选本次实际执行范围。

### 执行规则

`sync/execute` 必须显式接收本次选择的 `selected_sections`。

未被选择的 `section`：

- 不得执行
- 不得隐式连带写入

执行同步前，服务端必须再次校验目标环境 hash / preview 基线，保证执行时所依据的目标状态与预览时一致；不一致时必须拒绝执行，并要求重新预览。

### 依赖校验

服务端必须校验 `section` 依赖关系。

如果本次选择的 `section` 所依赖的前置数据在目标环境不存在或不兼容，则必须拒绝执行，并返回明确的依赖缺失信息。

### 删除与失效数据

删除项默认不执行，只有管理员显式勾选后才允许进入执行集。

被手动隐藏的渠道及其相关下游配置，必须被视为完全失效数据：

- 不进入配置快照
- 不参与同步预览
- 不参与同步执行
- 不进入客户端最终配置

## 支付路由匹配与唯一性规则

支付路由按“游戏 + 支付方式”维度维护优先级链路。

每条支付路由可包含以下匹配条件：

- `package`
- `channel`
- `market`
- `country`
- `currency`

其中空字段统一表示 `*`，即通配任意值。

### Market 匹配语义

- `CN`：仅匹配中国大陆市场
- `JP / KR / SEA / HMT`：仅匹配对应具体市场
- `GLOBAL`：仅作为海外默认兜底市场，不匹配 `CN`
- `*`：匹配任意市场

因此：

- 目标市场为 `CN` 时，只允许命中 `CN` 或 `*`
- 目标市场为 `JP / KR / SEA / HMT` 时，允许命中具体 `market`、`GLOBAL` 或 `*`
- 目标市场为 `GLOBAL` 时，只允许命中 `GLOBAL` 或 `*`

### 候选路由命中规则

候选路由必须满足：

- `package` 相等或为 `*`
- `channel` 相等或为 `*`
- `market` 按上述规则可匹配
- `country` 相等或为 `*`
- `currency` 相等或为 `*`

### 候选路由排序规则

命中多条路由时，按以下顺序决策：

1. `package` 精确命中优先于 `package = *`
2. 具体 `market` 优先于 `GLOBAL`
3. `GLOBAL` 优先于 `market = *`
4. 在其余条件相同的情况下，显式条件越多的路由优先级越高
5. 若仍可同时命中，则 `priority` 越小优先级越高

### 唯一性约束

为避免歧义，支付路由必须满足以下约束：

- 同一个游戏、同一个 `pay_way` 下，生效中的 `priority` 不允许重复
- 同一个游戏、同一个 `pay_way` 下，归一化后的选择器组合不允许重复

归一化后的选择器组合指：

- `package`
- `channel`
- `market`
- `country`
- `currency`

其中空字段必须按 `*` 参与唯一性比较。

如果新增或编辑路由后会造成冲突，服务端必须拒绝保存，并返回明确冲突信息。

## `GLOBAL` 与具体 Market 的配置合并规则

客户端最终配置按目标 `market` 生成，不同市场使用不同合并规则。

### 通用原则

- 游戏级配置先作为基础层
- 被隐藏的数据、无效数据、不兼容数据不得参与合并
- `market` 级实例的覆盖是实例级覆盖，不是字段级深度合并
- 创建时允许从其他 `market` 复制初始值，但创建后不再联动

### CN 市场

当目标市场为 `CN` 时：

- 只加载游戏级配置
- 只加载 `CN` 市场下的有效实例
- 不加载 `GLOBAL` 市场实例

### 具体海外市场

当目标市场为 `JP / KR / SEA / HMT` 时：

- 先加载游戏级配置
- 再加载 `GLOBAL` 市场下的有效实例
- 最后加载当前具体市场下的有效实例
- 若具体市场与 `GLOBAL` 存在同一渠道实例，则具体市场实例整体覆盖 `GLOBAL` 实例

### GLOBAL 市场

当目标市场为 `GLOBAL` 时：

- 只加载游戏级配置
- 只加载 `GLOBAL` 市场下的有效实例

### 覆盖范围

若具体市场与 `GLOBAL` 同时存在同一逻辑对象，则按以下规则处理：

- 同渠道实例：具体市场覆盖 `GLOBAL`
- 同渠道包实例：具体市场覆盖 `GLOBAL`
- 同登录配置实例：具体市场覆盖 `GLOBAL`
- 同 IAP 配置实例：具体市场覆盖 `GLOBAL`
- 同支付路由：具体市场优先于 `GLOBAL`

覆盖必须以完整实例为单位，不允许从 `GLOBAL` 继承部分字段，再与具体市场实例做字段级拼接。

## 前端页面交互与状态展示规则

### 游戏详情页的信息架构

游戏详情页分为两层：

#### 游戏级区域

负责展示和编辑以下全局配置：

- 游戏基础信息
- 已启用市场集合
- 法务链接
- 自有账号认证
- 商品
- 收银台模板绑定
- 支付路由
- 配置快照
- 同步记录

#### 渠道实例区域

负责展示和编辑按 `market` 维度拆分的渠道实例配置。

该区域默认展示当前游戏下所有 `market` 的所有 `GameMarketChannel` 实例，而不是只展示单个 `market`。

页面必须提供以下筛选能力：

- `market`
- 渠道名
- 兼容状态
- 隐藏状态
- 配置状态

其中 `market` 默认值为 `全部`。

### 渠道实例列表

渠道实例列表中，每一行都代表一个独立的 `GameMarketChannel` 实例。

列表必须展示：

- `market`
- 渠道名称
- 国内/非国内属性
- 兼容状态
- 隐藏状态
- `config_status`
- 是否进入最终配置
- 最近更新时间

同一逻辑渠道在不同 `market` 下必须分开显示，例如：

- `GLOBAL / Google`
- `JP / Google`

### 新增渠道交互

新增渠道时，必须先选择目标 `market`，再展示该 `market` 下允许选择的渠道集合。

规则如下：

- `market = CN` 时，只显示国内渠道
- `market != CN` 时，只显示非国内渠道
- 已隐藏渠道默认不出现在新增可选列表中
- 当前游戏下同一 `market` 已存在的渠道不得重复新增

新增时支持两种初始化方式：

- 空白创建
- 从其他 `market` 的同渠道复制初始值

如果选择复制初始值，则：

- 仅复制普通字段
- `secret` 字段必须清空
- `file` 字段必须清空
- 新实例初始 `config_status = invalid`
- `last_check_message` 必须提示缺少必填敏感字段或文件字段

### 不兼容与隐藏展示

如果某个渠道实例因市场规则变化而与当前 `market` 不兼容：

- 不自动删除
- 在列表中保留
- 行内标红
- 明确提示“不兼容当前 market”
- 允许管理员手动隐藏

隐藏后的行为必须固定为：

- 默认不出现在主列表
- 可通过“显示隐藏项”筛选查看
- 不进入配置快照
- 不参与同步预览
- 不参与同步执行
- 不进入客户端最终配置

页面必须清楚区分“不兼容”和“已隐藏”：

- 不兼容：表示当前规则下不可继续正常使用，但仍保留记录
- 已隐藏：表示管理员明确将该实例移出运行时生效集合

### 配置状态展示

所有模板驱动配置页面都必须明确显示 `config_status`，且不得隐藏异常状态。

当前固定状态为：

- `empty`
- `invalid`
- `valid`

页面含义如下：

- `empty`：尚未建立有效配置
- `invalid`：已存在配置结构，但缺少必填字段、敏感字段、文件字段，或未通过校验
- `valid`：配置完整且通过校验

如果实例是通过复制创建，且 `secret` / `file` 被清空，则必须显示为 `invalid`，不能显示为 `empty`。

### 运行态标识

凡是会影响配置快照、同步、客户端最终配置的渠道实例，都必须明确显示运行态标识。

建议至少提供以下只读标记：

- `Included in Snapshot`
- `Included in Sync`
- `Included in Runtime Config`

当实例被隐藏、不兼容或配置无效时，上述运行态标记必须同步变为不可生效状态，并给出明确原因提示。

## 建议写回原架构文档的重点

建议优先把以下内容写回原始架构文档：

- `go_domain_api_draft.md`：补足 `market` 模型、`GameMarketChannel`、支付路由、同步接口语义
- `backend_agent_execution.md`：补足渠道兼容性校验、模板版本化、`section` 级同步执行规则
- `frontend_agent_execution.md`：补足全 market 总览、按 market 过滤、隐藏/不兼容/复制创建/运行态标识规则

在后续实现计划中，建议把“渠道按 market 拆分”视为优先级最高的结构性决策，因为它会同时影响：

- 数据模型
- HTTP API
- 同步快照结构
- 前端信息架构
- 客户端最终配置生成逻辑
