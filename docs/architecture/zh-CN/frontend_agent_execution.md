# 前端 Agent 执行清单

这份清单给前端 agent 使用，默认基于 `pure-admin-thin` 的薄壳思路推进。

## 目标

构建一个简洁、长期可维护的发行管理后台前端，对接后端 DTO，并把复杂业务拆到清晰页面结构里。

## 阶段 0：清理壳子并初始化

- 基于 `pure-admin-thin` 思路建立壳子
- 去掉 demo 页面、mock、展示型示例
- 保留：
  - 登录壳
  - 路由守卫
  - 菜单
  - Tabs
  - 上传组件
  - 权限指令/权限组件
  - 表格与表单基础件
- 顶部常驻环境标识

完成标准：
- 前端能启动
- 没有残留 demo 菜单
- 全局可见当前环境

## 阶段 1：路由与状态层

- 建立主路由：
  - `/dashboard`
  - `/games`
  - `/channels`
  - `/cashier`
  - `/audit`
  - `/system`
- 建立 Pinia store：
  - `auth`
  - `permission`
  - `app`
  - `dictionary`

完成标准：
- 路由结构稳定
- 状态层可支撑后续页面

## 阶段 2：设计系统和通用页面容器

- 建立：
  - 列表页容器
  - 详情页容器
  - 抽屉表单
  - 差异抽屉
  - 状态标签
  - 密文字段展示
- 统一设计 token：
  - 颜色
  - 间距
  - 字号
  - 圆角
  - 阴影

完成标准：
- 样式改动主要集中在通用层
- 页面结构一致

## 阶段 3：游戏管理

- 建：
  - 游戏列表
  - 新建游戏
  - 游戏详情
  - 市场 Tab
  - 法务链接 Tab
- 清楚展示：
  - `game_id`
  - `game_secret`
  - 默认市场

完成标准：
- 游戏主数据前端可用

## 阶段 4：渠道与渠道包

在游戏详情里建立 Tab：

- 渠道
- 渠道包
- 商品
- 自有账号认证
- 渠道登录
- IAP
- 收银台
- 同步记录

完成标准：
- 新增渠道时能清楚看到该渠道的登录/支付策略
- 包管理可在详情页闭环完成

## 阶段 5：自有账号认证与渠道登录

- 自有账号认证配置页面
- 渠道强制登录配置页面
- 表单全部由模板驱动
- 配置状态要明确显示：
  - `empty`
  - `invalid`
  - `valid`

完成标准：
- 只加了认证方式但没填参数，页面能直接警告

## 阶段 6：商品与 IAP

- 商品列表与编辑
- 包级商品映射编辑
- 清楚区分：
  - `product_id`
  - `product_id_override`
  - `price_id`
  - `price_id_override`
- IAP 配置和包级覆盖

完成标准：
- 前端不会把 IAP 商品 ID 和收银台价格档混淆

## 阶段 7：收银台模板与汇率审核

- 顶层收银台页面：
  - 模板列表
  - 模板详情
  - 版本列表
  - 价格矩阵
  - 汇率同步审核
- 游戏详情里的收银台 Tab：
  - 绑定模板
  - 模板版本
  - 游戏级价格覆盖

完成标准：
- 运营能看清模板和游戏覆盖的边界
- 汇率变更默认需要人工确认

## 阶段 8：支付路由

- 页面支持：
  - 支付方式
  - 支付提供商
  - 公司主体
  - 商户账户
  - 游戏支付路由
- 路由要能直观看到优先级和兜底关系

完成标准：
- 能在页面里把某个支付方式切到另一个 PSP

## 阶段 9：配置快照与同步

- 游戏详情里支持：
  - 配置快照列表
  - JSON 预览
  - JSON 下载
  - 发布
  - `Sync to Production`
- `sandbox` 环境才显示同步动作
- 差异抽屉必须高亮：
  - 新增
  - 修改
  - 删除

完成标准：
- 同步动作安全、清楚、可确认

## 阶段 10：审计与打磨

- 审计日志页
- 完整空态、异常态、权限不足态
- 收窄危险操作视觉样式
- 保证窄屏下仍可用

## 前端必须遵守的规则

- 不要把后台登录 UI 和玩家登录配置 UI 混在一起
- 不要隐藏 invalid/empty 配置状态
- 尽量用抽屉，不要到处跳新页面
- 所有模板驱动表单用统一渲染机制
- 环境标识始终可见
- `production` 视图里不允许出现可执行的 `Sync to Production`
